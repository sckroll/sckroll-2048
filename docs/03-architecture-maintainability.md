# 아키텍처와 유지보수성 감사

## 종합 판정

현재 구조는 소규모 학습 프로젝트의 맥락과 기능 흐름을 파악하기 쉬운 편임.
v2 기능을 같은 클래스에 추가할 경우 변경 범위와 회귀 위험이 빠르게 커질 구조임.
TypeScript 전환의 핵심 과제는 문법보다 상태 소유권과 계층 경계 확립임.

## 현재 의존 흐름

~~~text
index.html
  └─ main.js
      └─ App
          ├─ Info
          ├─ Actions
          ├─ Log
          ├─ Popup
          └─ Board
              ├─ Block
              ├─ config
              └─ utils
~~~

<code>App</code>이 화면 객체 생성과 콜백 연결을 담당하는 조립 지점임.
화면 클래스 간 직접 import가 제한된 점은 유지할 가치가 있는 특성임.
브라우저 전역과 상태 저장소의 실제 의존은 구조도보다 넓은 상태임.
위 그림은 핵심 생성 흐름만 표현하며 <code>App</code>·<code>Actions</code>·<code>Info</code>·<code>Log</code>의 공유 <code>config</code>·<code>utils</code> 의존은 생략한 구성임.

## 책임 집중

<code>Board</code>가 동시에 맡는 책임은 다음과 같음.

- 게임 보드와 타일 상태 보관
- 빈 위치와 최대 타일의 파생 상태 보관
- 이동·병합·점수·승패 규칙
- 새 타일 생성 호출
- 보드와 타일 DOM 생성·변경·삭제
- 키보드·마우스·터치 입력 해석
- 최고 점수 로컬 저장소 접근
- 로그와 정보 화면 콜백 호출
- 시작·중지·계속하기 플래그 관리

<code>js/Board.js:38-573</code>에 이 책임이 집중된 상태임.
Canvas 교체나 undo/redo 추가가 게임 규칙까지 흔들 수 있는 주된 원인임.

## 상태 소유권

| 표현 | 내용 | 문제 |
|---|---|---|
| <code>state</code> | 숫자 행렬 | 실질적인 원본 |
| <code>emptyPos</code> | 문자열 좌표 목록 | 행렬과 중복 |
| <code>largestNum</code> | 최대 타일 | 행렬에서 계산 가능한 값 |
| DOM 클래스 | <code>r0</code>, <code>c0</code>, 색상 | 위치·값의 세 번째 표현 |
| <code>prevState</code> | 직전 행렬 | 한 턴 뒤 덮어쓰기 |
| 로그 <code>dataset</code> | JSON 이전 행렬 | 화면 요소가 이력 저장소 역할 수행 |
| 플래그 | 중지·계속·포인터 상태 | 도메인과 입력 상태 혼재 |

부분 실패나 비동기 애니메이션 도입 시 표현 사이의 불일치 가능성이 큼.
v2에서는 불변 <code>GameState</code> 하나를 원본으로 두고 화면을 투영하는 구조가 필요함.

## 결합과 부작용

- <code>new Block</code>이 난수 선택, 빈 좌표 제거, 행렬 변경, DOM 추가를 함께 수행
- <code>Board.update</code>가 점수 계산, DOM 변경, 빈 좌표 변경, 로그 발행을 함께 수행
- <code>Actions</code>와 <code>utils</code>가 로컬 저장소와 미디어 쿼리에 직접 접근
- 입력 이벤트가 <code>window</code>에 등록되고 제거 경로가 없는 상태
- 콜백 setter 호출 순서에 의존하는 암묵적 초기화 계약
- 방향을 문자열 조합으로 전달하고 완전성 검사가 없는 상태
- 설정 파일에 규칙, 문구, 외부 링크, 저장소 키가 함께 배치된 상태

각 생성자의 부작용이 커서 독립 단위 테스트와 객체 재사용이 어려운 구조임.

## TypeScript 전환 시 드러날 문제

| 유형 | 현재 사례 | v2 처리 |
|---|---|---|
| nullable DOM | <code>getElementById</code>, <code>querySelector</code> 즉시 사용 | 루트 검증과 요소 참조의 명시적 소유 |
| 혼합 저장 타입 | 최고 점수의 문자열·숫자 혼용 | 저장 경계의 파싱·검증 |
| 미선언 필드 | 생성자와 setter에서 동적 생성 | 필드 선언과 필수 의존성 주입 |
| 선택 콜백 | 설정 전 호출 가능성 | 이벤트 포트 또는 기본 무동작 구현 |
| 방향 문자열 | 조건 분기 누락 가능성 | 리터럴 유니온과 완전 분기 |
| 동적 객체 매핑 | 배열 순서 기반 <code>MoveData</code> | 타입이 있는 객체 리터럴 |
| 인덱스 접근 | 행렬 전반의 존재 가정 | <code>noUncheckedIndexedAccess</code> 적용 |
| 이벤트 대상 | <code>target</code>의 요소 타입 가정 | <code>currentTarget</code>과 구체 타입 |
| JSON 데이터 | 로그와 저장소의 무검증 파싱 | 버전과 런타임 검증 |
| 난수 | <code>Math.random</code> 고정 | <code>RandomSource</code> 주입 |

## 테스트 가능성

현재 저장소에는 <code>package.json</code>, 테스트 실행기, 테스트 파일, CI 구성이 없음.
<code>Board</code> 생성에는 DOM과 로컬 저장소가 필요함.
이동 함수도 인스턴스 필드와 이동 이벤트 목록을 직접 변경함.
난수 재현 수단이 없어 생성 타일까지 포함한 시나리오 테스트가 어려운 상태임.

리팩터링 전 브라우저 특성 테스트와 기준 보드 대조 테스트가 우선 필요함.
새 엔진은 브라우저 없이 실행 가능한 단위 테스트를 핵심 완료 조건으로 삼아야 함.

## 신규 작업자 관점의 장점

- README의 기능 설명과 화면 이미지
- 구조도와 게임 흐름도
- 이동 방향 정규화 이미지
- 병합의 기준·검사 단계 이미지
- 클래스명과 파일명의 비교적 직접적인 역할 표현
- 한국어 주석과 JSDoc
- 복잡한 빌드 절차가 없는 현재 실행 방식

## 신규 작업자 관점의 장애

- 공식 로컬 실행·검사·배포 절차 부재
- 행렬, 빈 좌표, DOM 사이의 동기화 불변식 설명 부재
- 이동 이벤트 두 개가 병합 하나를 표현하는 순서 계약 부재
- <code>MoveData</code>가 <code>Board.js</code> 내부에만 선언된 상태
- <code>Log</code>의 클래스 설명이 상단 정보 컨테이너로 잘못 기록된 주석
- <code>Popup</code> 두 번째 콜백 설명의 복사 오류
- 최고 점수 저장 시점과 테마·로그 저장 정책의 문서화 부재
- README 흐름도와 실제 마우스 입력 지원의 차이
- 회귀 테스트 부재에 따른 수정 결과 확인 부담

## 빌드와 배포 위험

- 브라우저가 원본 JavaScript를 직접 로드하는 제로 빌드 구성
- 외부 reset CSS, Google Fonts, Font Awesome에 대한 런타임 CDN 의존
- 네트워크 차단, CDN 장애, CSP 강화 시 표시 품질 저하 가능성
- TypeScript 소스와 배포 산출물을 구분할 디렉터리 규칙 부재
- 잠금 파일과 실행 환경 버전 정책 부재
- 정적 분석, 타입 검사, 테스트, 프로덕션 빌드의 자동 검증 부재

## 유지보수 원칙

- 보드 상태를 단일 진실 원천으로 취급
- 빈 셀과 최대 타일을 파생값으로 계산
- 성공 이동 한 번을 하나의 원자적 상태 전이로 취급
- 정적 렌더에 <code>GameState</code>, 턴 애니메이션에 <code>TurnResult</code>만 전달
- 로그 화면을 이력 데이터의 읽기 전용 투영으로 구성
- 생성자에서 외부 상태 변경 최소화
- 모든 이벤트 등록에 대응하는 해제 수명주기 제공
- 저장 데이터에 버전·검증·기본값 복구 제공
- 새 구조의 파일 역할과 금지 의존을 루트 <code>AGENTS.md</code>에 유지
