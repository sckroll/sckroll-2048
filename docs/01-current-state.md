# 현재 상태와 종합 판정

## 프로젝트 개요

- Vanilla JavaScript와 브라우저 ES 모듈로 구현된 4×4 2048 클론
- 키보드, 마우스 드래그, 터치 스와이프 입력 지원
- 점수, 최고 점수, 턴, 라이트·다크 모드, 이동 로그 제공
- 2048 달성 후 계속하기와 게임 오버 대화상자 제공
- 프레임워크, 빌드 도구, 패키지 관리자 설정이 없는 정적 사이트 구성
- Netlify 배포 주소가 README에 기록된 상태

## 저장소 구성

| 경로 | 역할 | 규모와 특징 |
|---|---|---|

| <code>images</code> | README 설명 자료 | 화면 6개, 구조·흐름 2개, 알고리즘 9개 |

## 현재 실행 흐름

1. <code>main.js</code>에서 <code>App</code> 생성
2. 테마 판정 후 시작 오버레이 표시
3. 시작 시 정보, 로그, 액션, 보드 객체 생성
4. 입력 직전 <code>prevState</code> 복사
5. 방향별 행렬 변환 후 오른쪽 기준 이동·병합 수행
6. DOM 타일, 빈 좌표, 점수, 최고 점수, 턴, 로그 갱신
7. 승리 판정·콜백 후 새 타일 생성, 이어서 게임 오버 판정

README의 <code>structure.png</code>는 <code>App</code> 중심의 의존 관계를 잘 표현함.
README의 <code>flow.png</code>는 시작부터 승패까지의 주요 순환을 표현함.
실제 구현의 마우스 입력, 전역 이벤트 수명주기, 상태 소유권은 두 이미지에서 생략된 상태임.

## 현재 구현의 강점

- 모듈 간 순환 import 부재
- 단일 <code>App</code> 조립 지점을 통한 화면 객체 연결
- 작은 파일 수와 평평한 디렉터리에 따른 초기 탐색 용이성
- 다수 메서드의 한국어 JSDoc과 주석
- README의 화면, 구조, 흐름, 이동·병합 알고리즘 설명
- 네 방향을 하나의 행 처리 방식으로 통일한 아이디어
- 고정 4×4 범위에서 충분히 작은 계산량
- 라이트·다크 모드와 데스크톱·모바일 플레이 제공
- 원작과 구분되는 이동 로그 기능

## v2 준비도 판정

현재 버전의 일반적인 4×4 플레이 유지에는 큰 계산상 장애가 없는 상태임.
TypeScript 문법 변환만으로는 Canvas, undo/redo, 가변 목표의 안정적 수용이 어려운 구조임.
순수 게임 엔진과 완전한 상태 스냅샷을 먼저 확보한 뒤 UI를 교체하는 순서가 적합함.
기존 구현은 폐기 대상보다 특성 테스트와 UX 비교를 위한 기준 구현으로 활용할 가치가 큼.

## 구현 전 확정 필요 항목

- 새 타일의 2 대 4 확률: 현행 50 대 50 또는 관례적인 90 대 10
- 사용자 지정 목표의 최댓값
- 이력 최대 길이와 오래된 항목 제거 안내
- v1 저장값의 승계 범위
