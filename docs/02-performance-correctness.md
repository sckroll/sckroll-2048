# 성능과 정확성 감사

## 요약 판정

고정 4×4의 이동·병합 계산은 현재 체감 성능에 문제가 없는 수준임.
일반 이동 결과도 기준 구현 대조에서 불일치가 발견되지 않은 상태임.
주요 성능 위험은 계산보다 장시간 누적되는 로그 DOM과 이벤트 리스너임.
주요 정확성 위험은 승패 동시 처리와 입력 세션 초기화 누락임.

## 이동·병합 방식

- 방향에 따라 행렬을 전치하거나 각 행의 원소 순서를 반전하는 정규화
- 모든 방향을 오른쪽 이동 형태로 바꾼 뒤 행 단위 처리
- 오른쪽 기준 셀과 왼쪽 검사 셀의 중첩 순회
- 빈 셀 건너뛰기, 같은 값 병합, 다른 값 경계 처리
- 처리 후 역변환을 통한 원래 좌표계 복원
- 이동 이벤트 두 종류를 통한 삭제 타일과 결과 타일 표현

관련 구현은 <code>js/Board.js:283-388</code>과 <code>js/utils.js:10-35</code>임.
README의 이동 이미지 4개와 병합 이미지 5개가 이 정규화와 기준·검사 순서를 설명함.

## 계산 복잡도

| 단계 | 일반화 복잡도 | 고정 4×4 영향 |
|---|---|---|
| 행렬 전치·반전 | O(R×C) | 최대 16개 셀 처리 |
| 행 단위 중첩 병합 | O(R×C²) | 행마다 최대 6개 비교 수준 |
| 빈 좌표 <code>findIndex</code> | 이동 이벤트마다 O(R×C) | 목록 최대 16개 |
| DOM 타일 <code>querySelector</code> | 이동 이벤트마다 DOM 탐색 | 타일 최대 16개 |
| 게임 오버 검사 | O(R×C) | 인접 셀 검사 |

정사각형 N×N 일반화 기준의 핵심 이동 계산은 O(N³)임.
4×4에서는 상수 크기에 가까워 최적화 우선순위가 낮은 영역임.
가변 보드 크기까지 지원할 경우 압축·단일 병합 순회의 O(N²) 엔진이 적합함.

## 정확성 검사

- 4개 값으로 구성한 단일 행 256개와 네 방향 조합 1,024건 전수 대조
- 6개 값으로 구성한 무작위 전체 보드 50,000개와 네 방향 조합 200,000건 대조
- 기준 압축·병합 구현과 결과 행렬 비교
- 대조 결과 불일치 0건
- <code>[2,2,2,2]</code>, <code>[2,2,2,0]</code>, 다중 병합의 한 턴 재병합 금지 확인
- 별도 무작위 검사에서 <code>emptyPos</code>와 실제 빈 셀 집합의 일반 경로 일치 확인
- 저장소에 남기지 않은 일회성 검사라는 한계

## 일회성 계산 측정

| 항목 | 결과 |
|---|---|
| 실행 횟수 | 100,000회 |
| 전체 시간 | 191.38ms |
| 이동당 시간 | 약 1.91µs |
| 포함 범위 | 행렬 변환, 병합, 이동 데이터 생성 |
| 제외 범위 | DOM 갱신, 로그, 애니메이션, 화면 합성 |

개발 환경과 런타임에 종속된 상대 참고값임.
v2 순수 엔진의 회귀 기준은 동일 장비·동일 데이터셋 비교가 적합함.

## 로그 성능

성공한 매 턴마다 <code>Log.add</code>가 다음 자원을 추가함.

- <code>.log-item</code>과 하위 DOM 노드
- 마우스 진입과 이탈용 이벤트 리스너 2개
- 이전 4×4 행렬의 JSON 문자열
- 방향, 턴, 점수 표시용 노드

로그를 닫아도 <code>$logContent</code>가 메모리에 남아 새 항목을 계속 수용함.
화면 연결 여부와 데이터 누적 여부가 분리되지 않은 결과임.
턴 수를 T, 보드 한 변을 N으로 볼 때 저장 공간은 O(T×N²)임.
고정 4×4에서도 높은 목표 타일의 긴 세션은 T에 비례한 DOM·리스너 증가를 유발함.

툴팁은 마우스 진입 시 16개 미니 셀을 생성하고 이탈 시 제거함.
한 번의 비용은 작으나 hover 전용 입력과 반복 DOM 생성의 UX·구조 부담이 존재함.
초기화 시 로그 노드를 하나씩 제거하는 비용도 O(T)임.

## 확인된 정확성 결함

### 승리와 게임 오버의 동시 콜백

승리 판정 후 새 타일 생성과 게임 오버 판정이 계속되는 순서임.
다음과 같은 거의 가득 찬 보드에서 왼쪽 이동과 특정 생성 타일로 재현 가능함.

~~~text
1024 1024    2    4
   4    8   16    8
   8   16    8   16
  16    8   16    8
~~~

2048 병합 직후 유일한 빈 칸이 채워지면 승리·패배 콜백이 각각 호출될 가능성임.
<code>App</code>은 두 콜백에서 각각 새 오버레이를 생성하므로 중복 대화상자 위험이 큼.
한 턴의 이동, 생성, 터미널 판정, 상태 커밋을 원자적으로 처리할 필요가 있음.

### 짧은 드래그 뒤 좌표 잔존

임계값 50px 미만의 드래그에서 <code>handleSlide</code>가 좌표 초기화 전에 반환함.
이후 다른 위치의 클릭만으로 이전 목표 좌표와 새 시작 좌표가 조합될 가능성임.
예시는 100,100에서 110,100까지 짧게 이동한 뒤 200,100 위치를 클릭하는 흐름임.
포인터 세션의 시작·종료·취소마다 모든 좌표를 초기화할 필요가 있음.

### 최고 점수 유실

메모리 최고 점수는 병합 때 갱신되나 로컬 저장은 승리·패배 시점에만 수행됨.
새 기록 직후 재시작이나 새로고침 시 저장 전 값으로 돌아갈 가능성임.
조회 결과도 문자열 또는 숫자 0의 혼합 타입임.
정수 정규화와 갱신 즉시 영속화가 필요함.

## 방어성과 규칙 차이

- 빈 칸 없는 상태의 <code>createBlock</code> 호출 시 구조 분해 예외 가능성
- <code>emptyPos.findIndex</code> 실패 시 <code>splice(-1,1)</code>에 따른 마지막 좌표 손상 가능성
- <code>reverseRowElements</code>의 내부 행 제자리 반전에 따른 숨은 변이
- 새 타일 2와 4의 현행 확률이 각각 50%인 규칙
- 2048 관례의 90% 대 10%를 기대하는 사용자 경험과의 차이
- <code>largestNum === 2048</code>의 정확 일치 판정과 고정 목표
- 정사각형 보드에서만 맞는 행·열 상수 사용

## v2 성능 원칙

- 이동을 DOM 비의존 순수 함수로 구성
- 16셀 전체 Canvas 재그리기를 기본 전략으로 채택
- <code>requestAnimationFrame</code> 한 개로 애니메이션 수명주기 통합
- 이력을 직렬화 가능한 데이터로 저장
- 화면 로그에 이벤트 위임과 가상 목록 적용
- undo/redo의 커서 이동을 O(1)로 유지
- 빈 좌표와 최대 타일을 보드에서 계산하는 파생값으로 처리
- 난수 공급원 주입을 통한 재현 가능한 테스트 구성
- 프로파일링 근거가 생기기 전 <code>OffscreenCanvas</code>와 Worker 도입 보류

## 우선 회귀 테스트

- 네 방향의 연속 빈 칸, 단일 병합, 이중 병합, 병합 차단
- 이동 불가 입력의 보드·점수·턴·로그·난수 불변
- 이동·병합 단계의 타일 합 보존과 완성 턴 합의 생성 타일 값만큼 증가
- 점수 증가량과 생성된 병합 타일 합계 일치
- 새 타일 생성 후 빈 셀 집합 일치
- 승리와 게임 오버 충돌 재현
- 짧은 포인터 이동과 취소 뒤 오입력 부재
- 손상된 최고 점수 저장값 복구
- 10,000개 이력의 메모리와 화면 노드 상한
