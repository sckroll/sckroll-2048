# v2 검증과 품질 게이트

## 목적

전환 단계마다 규칙, 상태, 화면, 접근성, 성능의 회귀를 같은 기준으로 판정하기 위한 문서임.
세부 구현 순서는 [전환 계획](./07-migration-validation-plan.md)을 기준으로 함.

## 테스트 계층

| 계층 | 대상 | 핵심 역할 |
|---|---|---|
| 단위 | 이동, 병합, 생성, 상태, 이력 | 빠른 브라우저 비의존 검증 |
| 속성 | 임의 보드 불변식 | 합 보존과 기준 구현 대조 |
| 통합 | 컨트롤러, 저장소, 렌더 요청 | 한 턴의 원자성 검증 |
| Canvas | 좌표, DPR, 애니메이션 종료 | 픽셀·상태 정합성 검증 |
| E2E | 입력, 대화상자, 목표, undo/redo | 사용자 흐름 검증 |
| 접근성 | 이름, 역할, 포커스, 대비 | 자동 검사와 키보드 수동 검사 |
| 시각 회귀 | 핵심 화면 크기와 테마 | 레이아웃·색상 변화 검증 |

## 필수 규칙 행렬

- <code>[2,2,2,2]</code>, <code>[2,2,2,0]</code>, <code>[4,4,8,8]</code>의 네 방향
- 연속 빈 칸과 서로 다른 타일의 압축
- 이동 불가 입력과 꽉 찬 보드
- 이동·병합 단계의 합 보존과 완성 턴 합의 생성 타일 값만큼 증가
- 병합 결과 합계와 점수 증가량 일치
- 한 타일의 한 턴 두 번 병합 금지
- 생성 난수 0, 경계 직전, 경계, 1 직전
- 유효 이동 뒤 생성 한 번
- 이동 불가 입력의 난수 호출 0회

## 필수 상태 행렬

- 2048·4096·8192·16384 목표 달성
- 사용자 목표의 최솟값·최댓값·2의 거듭제곱 검증
- 목표 달성과 게임 오버의 같은 턴 발생
- 승리·패배 직전과 직후의 undo/redo
- undo 뒤 새 이동과 redo 제거
- redo의 난수 호출 0회
- 이력 상한 초과와 오래된 항목 제거
- 최고 점수의 undo 제외와 단조 증가
- 손상된 저장 JSON과 범위 밖 목표
- v1 테마·최고 점수 변환

## 필수 화면 행렬

- DPR 1·2·3과 크기 변경
- 320×568, 320×480, 390×844, 568×320, 844×390, 1280×720 화면
- 라이트·다크·시스템 테마
- 모션 감소
- 데스크톱 이력 패널과 모바일 하단 시트
- 2048·4096·8192·16384 이상 타일
- 시작, 목표 달성, 게임 오버 대화상자
- 저장소 사용 불가와 Canvas 초기화 실패 안내

## 접근성 수동 흐름

1. 키보드만으로 시작
2. 목표 선택과 새 게임 확인
3. 방향키 이동
4. undo와 redo
5. 이력 패널 열기와 항목 세부 정보 확인
6. 목표 달성 대화상자 처리
7. 게임 오버에서 undo
8. 테마 변경
9. 포커스의 논리적 복귀 확인

모든 단계에서 포커스 가시성, 접근 이름, 상태 알림, 배경 스크롤 차단을 확인할 필요가 있음.

## 성능 수용 기준

- 순수 이동 벤치마크의 기준선 대비 2배 이내 유지
- 16타일 애니메이션 중 50ms 초과 장기 작업 부재
- 60Hz 환경의 애니메이션 프레임 p95 16.7ms 이내
- 활성 애니메이션 루프 한 개
- undo/redo 커서 이동 O(1)
- 10,000개 이력에서 화면 연결 행 100개 이하
- 10,000개 이력의 전체 힙 사용량 측정과 기준값 기록
- 로그 닫힘 상태에서 화면 DOM 증가 부재
- 크기 변경 중 불필요한 연속 backing store 재생성 억제
- 애니메이션 종료 뒤 예약 프레임 0개

## 주요 위험과 통제

| 위험 | 통제 |
|---|---|
| 엔진과 렌더러의 상태 분기 | 불변 상태와 단방향 렌더 계약 |
| 빠른 입력의 애니메이션 충돌 | 제한 큐와 단일 완료 경로 |
| redo의 난수 차이 | 완성 스냅샷 복원 |
| 큰 이력의 메모리 증가 | 10,000개 상한과 가상 목록 |
| 높은 타일의 UI 붕괴 | 동적 팔레트와 글자 크기 함수 |
| 모바일 높이 부족 | 높이 기반 레이아웃과 <code>100dvh</code> |
| 저장 스키마 손상 | 런타임 검증과 항목별 기본값 |
| TypeScript 도입 지연 | 전환 1단계의 엄격 모드 선행 |
| Canvas 접근성 손실 | 의미 있는 DOM 보조 계층 |

## 예상 개발 명령

도구 기반 추가 뒤 다음 역할의 명령을 제공할 필요가 있음.

- <code>npm run dev</code>: 로컬 개발
- <code>npm run typecheck</code>: TypeScript 검사
- <code>npm run test</code>: 단위·통합 검사
- <code>npm run test:e2e</code>: 브라우저 흐름 검사
- <code>npm run lint</code>: 정적 분석
- <code>npm run build</code>: 프로덕션 빌드
- <code>npm run preview</code>: 빌드 결과 확인

현재 저장소에는 위 명령을 제공하는 <code>package.json</code>이 없는 상태임.
