# UX/UI 감사와 새 디자인 방향

## 감사 자료

- 데스크톱 라이트 화면 <code>desktop.png</code>
- 데스크톱 다크 화면 <code>darkmode.png</code>
- Android 세로 화면 <code>android.png</code>
- iOS 세로 화면 <code>ios.jpg</code>
- 열린 로그와 이전 보드 툴팁 <code>log-1.png</code>, <code>log-2.png</code>
- 1280×720, 390×844, 375×667, 844×390 로컬 실행 결과
- <code>index.html</code>, <code>css/style.css</code>, <code>css/variables.css</code>

## 이미지에서 확인한 현재 경험

### 데스크톱

- 중앙의 360px 보드와 상·하단 고정 바 구성
- 점수·최고 점수·턴의 우측 정렬
- 큰 화면에서 보드 주변 여백이 과도하게 남는 구성
- 로그가 좌측 가장자리에 독립 패널로 표시되는 구성
- 로그와 현재 보드 사이의 시선 이동 거리가 긴 상태
- 보드보다 하단 출처·개발자 정보가 큰 면적을 차지하는 인상

### 모바일

- 제목, 세 지표, 보드의 기본 계층은 비교적 명확한 구성
- 75vw 보드로 세로 화면의 가독성 확보
- 하단 버튼과 두 줄의 출처 정보가 게임 영역을 크게 잠식하는 구성
- Android 라이트 화면과 iOS 다크 화면에서 기본 플레이 가능성 확인
- 기기 높이와 안전 영역을 직접 반영하지 않는 고정 배치

### 로그

- 방향, 턴, 점수 증가량을 한 행에서 빠르게 확인 가능한 구성
- 이전 보드 미니어처를 통한 회고 기능의 차별점
- 아이콘 hover에만 의존하는 이전 보드 확인 방식
- 로그 패널이 게임 보드와 경쟁하는 큰 고정 높이
- <code>+0점</code> 반복 밀도와 미래 이력·현재 커서 시각 상태 부재

### 테마와 타일

- 라이트 모드의 청록 계열과 다크 모드의 남색 계열 일관성
- 빈 셀과 낮은 타일의 구분이 가능한 기본 명도 차
- 2048 이후 색상 클래스와 변수 부재
- 높은 숫자의 자릿수에 대응하는 글자 크기 규칙 부재
- 시스템 테마는 최초 판정에만 반영되는 구조

## 실행으로 확인한 레이아웃 문제

| 화면 | 결과 |
|---|---|
| 1280×720 | 보드 360×360, 상단 56px, 하단 약 73px로 기본 플레이 가능 |
| 390×844 | 열린 256×512 로그가 보드와 직접 겹침 |
| 375×667 | 로그를 닫은 상태의 보드·상단·하단 분리는 가능 |
| 844×390 | 보드 하단과 고정 푸터가 겹침 |
| 568×320 | 426px 보드가 화면 위로 벗어나 상단·하단과 겹침 |
| 320×480 | 보드가 상단과 약 8px, 하단과 약 19px 겹침 |

너비 중심 미디어 쿼리와 <code>100vh</code> 고정 셸이 낮은 가로 화면을 처리하지 못한 결과임.
<code>html { overflow: hidden; }</code> 때문에 겹친 콘텐츠로 이동할 스크롤 경로도 없는 상태임.
모바일 브라우저의 동적 주소창과 안전 영역에 대한 대응도 부재함.

## 입력 경험

- 방향키, 마우스 드래그, 터치 스와이프 지원
- 모든 마우스·터치 이벤트가 <code>window</code>에 등록된 상태
- 헤더 드래그만으로 실제 턴 증가가 재현된 보드 밖 입력
- <code>touch-action: auto</code>에 따른 브라우저 제스처와의 경쟁 가능성
- 방향키의 기본 동작 제어 부재
- 50px 고정 임계값에 따른 화면 밀도·보드 크기 비례 부족
- 짧은 드래그 뒤 목표 좌표가 남는 오입력 결함
- 애니메이션 중 연속 입력의 큐·잠금 정책 부재
- 조작 안내와 키보드 단축키 안내 부재

v2 입력은 보드 영역의 Pointer Events와 포인터 캡처를 중심으로 통합할 필요가 있음.
스와이프 임계값은 보드 한 변과 CSS 픽셀의 조합으로 계산하는 방식이 적합함.

## 애니메이션

- 새 타일의 총 300ms <code>fadeInDelay</code>와 전반 150ms 투명 구간
- 모든 타일 속성에 적용되는 200ms <code>transition: all</code>
- 화면 영역의 300ms 페이드
- 이동 위치 전환과 생성 페이드는 존재하나 병합 고유 강조가 없는 상태
- 병합 순간의 강조와 점수 증가 피드백 부재
- <code>prefers-reduced-motion</code> 대응 부재
- 빠른 연속 입력과 CSS 전환의 일관된 완료 시점 부재

v2에서는 이동, 병합, 생성의 전환 데이터를 엔진 결과에서 분리해 제공할 필요가 있음.
Canvas 애니메이션은 단일 시간축과 단일 <code>requestAnimationFrame</code> 루프로 관리하는 구성이 적합함.

## 접근성

### 확인된 문제

- 시작·승리·패배 팝업의 <code>role="dialog"</code>와 <code>aria-modal</code> 부재
- 팝업 열림 시 시작 버튼 포커스 이동과 닫힘 후 포커스 복귀 부재
- 점수·턴 변화의 <code>aria-live</code> 알림 부재
- 보드 상태를 읽을 수 있는 의미 구조 부재
- 로그 이전 보드 버튼의 키보드 포커스와 이름 부재
- GitHub 아이콘 링크의 접근 가능한 이름 부재
- 명시적인 <code>:focus-visible</code> 스타일 부재
- 전체 요소의 텍스트 선택 차단
- 숫자 외의 보조 구분이 유사한 색상 계열에 집중된 구성
- 테마 버튼의 현재 상태를 전달하는 <code>aria-pressed</code> 부재
- 로그 열림 상태를 전달하는 <code>aria-expanded</code>와 <code>aria-controls</code> 부재
- 다크 설명문과 배경의 약 4.46:1 대비로 일반 텍스트 4.5:1 기준에 근소하게 미달

### v2 기준

- HUD와 버튼을 의미 있는 DOM으로 유지
- Canvas를 장식 렌더링 계층으로 취급
- 포커스 가능한 게임 영역과 숨김 보드 표 또는 상태 요약 제공
- 이동 뒤 점수 증가와 최대 타일을 짧게 알리는 정중한 live region 제공
- 대화상자의 포커스 트랩, 제목 연결, 복귀 대상 관리
- 모든 아이콘 버튼의 한국어 접근 이름 제공
- 최소 44×44 CSS 픽셀의 조작 영역
- 일반 텍스트 4.5:1, 큰 텍스트와 비텍스트 요소 3:1 대비 목표
- 모션 감소 설정에서 전환 시간 0 처리

## 정보 구조 문제

- 한국어 로그와 영어 조작 버튼의 혼합, Poppins 한글 글리프 부재에 따른 서체 불일치
- 목표 타일 표시와 변경 진입점 부재
- 최고 점수의 목표별 의미 부재
- 재시작이 즉시 실행되는 파괴적 동작
- undo/redo 가능 여부와 현재 이력 위치를 보여줄 영역 부재
- 로그 열림 상태가 아이콘 방향에만 의존
- 출처 정보가 핵심 조작과 같은 시각 계층에 배치된 상태

## v2 화면 구조

### 데스크톱

~~~text
상단: 제목 · 목표 선택 · 테마/설정
본문: 상태 카드 + Canvas 보드 | 이력 패널
하단: 되돌리기 · 다시 실행 · 새 게임 · 조작 안내
보조: 출처와 저장 상태
~~~

- 최대 너비가 있는 게임 셸과 보드 중심의 두 열 구성
- 이력 패널의 독립 스크롤과 현재 커서 강조
- 좁은 데스크톱에서 이력 패널 접기
- 출처 정보를 작은 보조 영역으로 이동

### 모바일

~~~text
상단: 제목 · 목표
지표: 점수 · 최고 점수 · 턴
본문: 정사각 Canvas 보드
도구: 되돌리기 · 다시 실행 · 새 게임
이력: 하단 시트
~~~

- <code>100dvh</code>와 안전 영역 inset 반영
- 가용 너비와 높이 중 작은 값에 맞춘 보드 크기
- 로그를 보드 위 고정 패널로 배치하지 않는 원칙
- 가로 화면에서 보드와 도구를 좌우로 재배치하는 높이 기반 쿼리

## 시각 디자인 방향

- 보드를 가장 강한 표면으로 두는 계층
- 카드 수를 제한한 간결한 대시보드 형태
- 목표 타일과 현재 최대 타일을 한 문맥에서 표시
- 타일 지수에 따른 토큰 또는 HSL 팔레트 생성
- 2048 이후에도 대비가 유지되는 알고리즘형 색상
- 숫자 자릿수에 따른 Canvas 글자 크기 단계
- 라이트·다크·시스템 테마의 세 상태
- 현재 상태, 과거 이력, redo 가능한 미래 이력의 명확한 색·선 구분

## 권장 모션 규격

| 동작 | 권장 시간 | 곡선과 효과 |
|---|---|---|
| 타일 이동 | 120~160ms | 감속 곡선 |
| 병합 강조 | 90~120ms | 1.0→1.12→1.0 크기 |
| 새 타일 생성 | 100~140ms | 0.75→1.0 크기와 투명도 |
| 점수 증가 | 160~220ms | 짧은 상승과 페이드 |
| 이력 이동 | 120~160ms | 커서와 보드의 동시 갱신 |
| 모션 감소 | 0ms | 최종 프레임 즉시 표시 |

애니메이션 중 입력은 최대 2개 방향의 제한 큐로 처리하는 방안을 권장함.
상태는 한 번만 커밋하고 렌더러는 전환 프레임만 보간하는 원칙이 필요함.

## UX 완료 기준

- 320px 너비와 200% 확대부터 데스크톱까지 겹침·가로 스크롤 부재
- 844×390과 유사한 낮은 화면에서 보드·도구 겹침 부재
- 열린 이력 화면이 보드 조작을 예기치 않게 가리지 않는 구성
- 키보드만으로 시작, 이동, undo, redo, 새 게임, 목표 변경 가능
- 터치에서 이전 보드와 이력 항목 확인 가능
- 모션 감소와 시스템 테마 변경 반영
- 목표 2048 초과 타일의 색상·글자 가독성 유지
- 승리·패배 대화상자 한 개만 표시
